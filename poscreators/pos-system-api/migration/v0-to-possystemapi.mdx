# Migration Guide: v0 SignatureCloud → POSSystemAPI (v2)

## Introduction

The legacy **v0 SignatureCloud API** (also referred to by its subdomain pattern `signaturcloud-sandbox.*`) is being superseded by the new **POSSystemAPI (v2)**. Migrating to the new API is required for:

- **E-invoicing support** — the v0 API does not support the new market-specific e-invoicing features.
- **Alignment with fiskaltrust Middleware v2** — the POSSystemAPI interface is designed to remain largely stable when the middleware transitions from v1.2 to v2, making this migration valuable preparation.
- **Long-term supportability** — v0 is considered deprecated; customers are encouraged to migrate as soon as possible.

This guide is primarily targeted at POS Creators integrating **cloud cashboxes** in the **AT, FR, and DE** markets.

:::note Local cashboxes

Local cashbox migration also requires updated configuration. This is currently handled through internal guidance (sandbox/admin only). Please refer to [Local Cashboxes](#local-cashboxes) at the end of this guide.

:::

---

## Prerequisites

Before starting the migration, make sure you have:

- An active **fiskaltrust account** (sandbox and/or production).
- An existing integration that successfully produces receipts against the v0 SignatureCloud API.
- Access to a **sandbox cashbox** for testing your migrated integration before going live.
- Familiarity with the **POSSystemAPI business case examples**, which serve as the reference for the new request format:
  [https://developers.fiskaltrust.eu/#/pos-system](https://developers.fiskaltrust.eu/#/pos-system)

## Step 1 — Change the Base URL

### Old URLs (per market, v0 SignatureCloud)

| Market | Sandbox URL |
|--------|-------------|
| AT     | `https://signaturcloud-sandbox.fiskaltrust.at` |
| FR     | `https://signaturcloud-sandbox.fiskaltrust.fr` |
| DE     | `https://signaturcloud-sandbox.fiskaltrust.de` |

### New URL (POSSystemAPI v2, market-agnostic)

| Environment | URL |
|-------------|-----|
| Sandbox     | `https://possystem-api-sandbox.fiskaltrust.eu/v2` |
| Production  | `https://possystem-api.fiskaltrust.eu/v2` |

:::info

The new endpoint is **market-agnostic** — the same base URL is used for AT, FR, and DE. The market is determined by the `ftCashBoxID` and queue configuration, not the URL.

:::

Replace all occurrences of the per-market v0 base URL in your integration with the single new URL above. Make sure to append `/v2` to the path.

## Step 2 — Migrate the `*Case` Values

The `ftReceiptCase`, `ftChargeItemCase`, and `ftPayItemCase` fields use **different numeric codes** in v0 and POSSystemAPI v2. You must remap all case values in your requests.

The authoritative reference for the new values is the [POSSystemAPI business case examples](https://developers.fiskaltrust.eu/#/pos-system).

## Step 3 — Update `ftReceiptCaseData`

### v0 Format

In the v0 API, `ftReceiptCaseData` was a **JSON-encoded string** placed directly in the field value:

```json
"ftReceiptCaseData": "{\"Code\":\"20\", \"Message\":\"Archivage fiscal de période\", \"Information\":\"\"}"
```

### POSSystemAPI v2 Format

In the new API, `ftReceiptCaseData` is a **JSON object** whose keys are market codes (`AT`, `FR`, `DE`, etc.). The value for each market key remains a JSON-encoded string, preserving backward compatibility of the inner payload structure:

```json
"ftReceiptCaseData": {
  "FR": "{\"Code\": \"20\", \"Message\": \"Archivage fiscal de période\", \"Information\": \"\"}"
}
```

:::note

- The **outer value** is now a JSON object keyed by market code — **not** a plain string.
- The **inner value** is still a JSON-encoded string (i.e., a stringified JSON object). This inner format is unchanged from v0.
- Multiple markets can be combined in the same object when a single receipt spans multiple market contexts.

:::

### Examples by Market

**Austria (AT)**

```json
"ftReceiptCaseData": {
  "AT": "{\"Code\": \"20\", \"Message\": \"Jahresbeleg\", \"Information\": \"\"}"
}
```

**France (FR)**

```json
"ftReceiptCaseData": {
  "FR": "{\"Code\": \"20\", \"Message\": \"Archivage fiscal de période\", \"Information\": \"\"}"
}
```

**Germany (DE)**

```json
"ftReceiptCaseData": {
  "DE": "{\"Code\": \"20\", \"Message\": \"Jahresabschluss\", \"Information\": \"\"}"
}
```

**Multi-market (combined)**

```json
"ftReceiptCaseData": {
  "AT": "{\"Code\": \"20\", \"Message\": \"Jahresbeleg\", \"Information\": \"\"}",
  "FR": "{\"Code\": \"20\", \"Message\": \"Archivage fiscal de période\", \"Information\": \"\"}"
}
```

:::warning

If `ftReceiptCaseData` is not needed for a particular receipt, pass an empty object (`{}`) or omit the field entirely. **Do not pass a bare empty string (`""`)** as the top-level value — that was valid in v0 but is no longer accepted at the outer level in v2.

The inner string value for a given market key **may** be an empty string (`""`) when no additional case data is required for that market.

:::

## Testing Checklist

After making the changes above, validate your integration against the sandbox environment:

- [ ] **Normal sale receipt** — Issue a standard cash sale receipt and verify a successful response from `https://possystem-api-sandbox.fiskaltrust.eu/v2`.
- [ ] **Refund / void receipt** — Issue a refund or void receipt and verify the response.
- [ ] **Fiscal period archive receipt** — Issue an archive receipt using `ftReceiptCaseData` with `"Code": "20"` in the market-block format and verify the response.
- [ ] **Zero receipt** — Issue a zero receipt and verify the response.
- [ ] **All responses succeed** — Confirm there are no `4xx` or `5xx` errors against the new base URL.
- [ ] **Cross-check with business case examples** — Verify your request and response structures against the [POSSystemAPI business case examples](https://developers.fiskaltrust.eu/#/pos-system).

:::tip

Use the [POSSystemAPI business case examples](https://developers.fiskaltrust.eu/#/pos-system) page as the primary reference for validating that your request structures are correct for each business case.
